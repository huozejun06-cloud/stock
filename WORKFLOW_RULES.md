# 用户工作流指令集 — 提升 Codex 工作质量的 Meta-Instructions

> 从完整对话历史中提取的关键工作模式、质量规则和审查流程。
> 这些是"告诉 Codex 怎么干活"的指令, 与项目业务代码无关。

---

## 一、核心工作框架：Multi-Agent Review

**首次给出时的完整原文**：
```
启动 Multi-Agent Review。
Agent：
  Project Manager     — 需求分析, 任务拆分, 节奏控制
  Architect            — 目录结构, 模块划分, 数据流, SOLID
  Backend Engineer     — API, 算法, 数据库, 异常处理
  Frontend Engineer    — UI, 响应式, 组件拆分
  Data Scientist       — 数据真实性, EDA, 聚类, 无 mock
  Quant Researcher     — 选股逻辑, 回测, 指标合理性
  Code Reviewer        — Bug, 空指针, 死代码, 规范
  Security Reviewer    — 路径注入, 越权, 输入合法性
要求：
  所有 Agent 必须先独立分析, 不得互相参考
  之后进入委员会讨论
  如果出现冲突：继续讨论, 直到形成一致意见
最终输出：
  ① Consensus（一致共识）
  ② Modification Plan（修改计划）
  ③ Risk（风险）
  ④ 是否值得修改
```
**后期追加**：每个 Agent 的输出必须全部写到对话里（用户选的是"展开"模式）。

## 二、Git Checkpoint 纪律

**原文**：
```
任何 Agent 都不得直接修改代码。
第一阶段：Review
第二阶段：Committee Debate
第三阶段：生成 Patch Plan
第四阶段：等待我确认
第五阶段：创建 Git Checkpoint
  git add .
  git commit -m "Checkpoint: Before Task X"
第六阶段：开始修改
任何修改必须：
  · 一次只改一个 Task
  · 修改完成：编译 + 运行 + API测试 + 前端测试全部通过才能进入下一 Task
  · 否则立即 git reset 到 Checkpoint
```
**用户反复强调**：
- "记得每一步都加上checkpoint"（出现3次）
- "以后每次回答把检查点也加上"

## 三、工作节奏与审批流程

用户确立的循环：

```
1. Codex 展示方案（详细信息）
2. 用户审查
3. 用户说 "ok"（确认）
4. Codex 执行
5. Codex 创建 Checkpoint（git commit）
6. Codex 检查 + 验证
7. Codex 展示下一个方案
   ↺ 回到步骤1
```

**关键节奏节点**：
- 「每次做完之后检查一遍没问题了开始给我发下一个详细信息我来确认」（Task 粒度确认）
- 「先确定一下d的细节」（先细化再行动）
- 「做每一步都要创建checkpoint」（安全网）

## 四、质量检查模式

### 4.1 自我批判
「这个方案你在自己检查一遍分析一遍」— Codex 必须先审查自己的方案再给用户看。

### 4.2 阶段审计
- 「检查一遍a」 — Phase A 完整审计
- 「现在把改过的从头到尾做一遍检查跑一遍数据看看有没有问题漏洞缺点」
- 「再检查一遍还有什么可以优化的也列出任务」

### 4.3 数据真实性验证
「检查一下数据是否是最新的, 这些数据是否真实准确, 指标计算是否造假等」
「如何判断是否骗我用了历史缓存」— 要求证明数据是实时的, 不是 mock 或缓存。

### 4.4 纯分析不修改
「只列出问题和机会不需修改代码」— 用户明确要求只看不动。

### 4.5 端到端验证
「跑一遍选股把具体详细信息发给我」— 完整运行 pipeline 并输出全部结果。

## 五、批判性思维要求

### 5.1 对比分析
「根据我现在有的程序去分析一下然后把对比列出来批判性思维」— 比较新旧方案。

### 5.2 方案审查
「分析一下这段内容」— 让 Codex 分析用户给出的文本（如 V3.1 规划）。

### 5.3 漏洞扫描
「全面看一遍里面的所有文件给我列出架构和数据分析的技术指标」

## 六、展示格式要求

### 6.1 改动效果说明
「改完之后的具体改动和效果是什么」— 每次修改必须展示 before/after 对比。

### 6.2 详细方案先行
「先给我看详细方案, 我确认后再执行」— 绝不跳过计划直接动手。

### 6.3 像素级 vs 功能级
用户最初要求"像素级对应映射", 经过讨论后调整为更准确的表述, 但展示的精细度要求一直保持。

## 八、边界条件（当对话上下文压缩时）

1. 第一时间读取 `PROJECT_STATUS.md`

## 九、方案三重审查纪律（2026-07-04 追加）

**用户原文**：
```
每个任务给我详细方案时你要用多agent分析复查三遍后再给我生成方案给我
```

执行细则：
1. **在每个 Task 的详细方案公布之前**，必须先用 Multi-Agent 模式进行三重审查
2. 三重审查流程：
   - 第1遍：Codex 自己独立分析方案的正确性、合理性和风险
   - 第2遍：模拟 Multi-Agent 各角色从各自视角独立审查该方案
     （Architect / Backend Engineer / Data Scientist / Quant Researcher / Code Reviewer / Security Reviewer）
   - 第3遍：委员会辩论，汇总各角色意见，只有达成共识后才能将方案展示给用户
3. 三重审查的输出必须包含：
   - ✅ 方案正确性确认 或 ⚠️ 发现的问题
   - Risk 风险评估
   - 最终结论：是否值得按此方案执行
4. 此规则适用于 Phase F 至 Phase Z **所有未来 Task 的方案输出**
5. F-1 已提前出了方案（此规则追加之前），按原计划直接执行

## 十、改后三重验证纪律（2026-07-04 追加）

**用户原文**：
```
改完之后也要从不同的角度复查审核验证3确定没有问题
```

执行细则：
1. **每个 Task 修改完成后**，必须进行三重验证才能进入下一个 Task
2. 三重验证流程：
   - 第1遍：语法/运行时验证 — 代码能跑、不报错、不 crash
   - 第2遍：Multi-Agent 功能验证 — 各角色从自己视角验证改动是否生效、副作用是否可控
   - 第3遍：端到端集成验证 — 运行关键路径（pipeline/API）确认整体不坏
3. 三重验证的输出必须包含：
   - ✅ 所有验证通过
   - 或 ❌ 问题列表（修复后才能继续）
4. 未通过验证的修改，**自动 git reset 到 Checkpoint**

## 十一、全响应三重审查纪律（2026-07-04 追加）

**用户原文**：
```
以后每一次回答你都要多agent分析复查三遍后再给我内容
```

执行细则：
1. **每个回答发送之前**，必须经过 Multi-Agent 三重审查
2. 三重审查流程：
   - 第1遍：内容准确性审查 — 信息是否正确、完整、无遗漏
   - 第2遍：多角色视角审查 — 从不同 Agent 角色角度检查回答是否全面
   - 第3遍：质量终审 — 格式是否清晰、是否回答了用户问题、是否包含必要细节
3. 此规则适用于 **所有类型的回答**，包括但不限于：
   - 状态汇报、问题分析、方案展示、代码解释
   - 简短回答、长文分析
4. 三重审查不需要每次都展开写出审查过程，但必须在内部完成

## 十二、零豁免条款（2026-07-04 追加）

**用户原文**：
```
每一次必须先给我方案细节确定后才可以改代码
每次回答前先分析复查再给出答案
```

执行细则（**无任何豁免**）：
1. **任何修改之前**，必须先出详细方案 → 等待用户确认 → 才能动手改代码
2. **任何回答发送之前**，必须经过内部 Multi-Agent 三重审查
3. **"小修""小改""顺手修"等理由不能豁免以上两条**
4. 违反此条款时，用户有权要求回退到上一个 Checkpoint

## 十三、前端开发三大铁律（2026-07-05 追加）

> 承袭 ui-ux-pro-max、impeccable、react-bits、Karpathy 编码规范的设计理念

### 13.1 UI 质感与规范硬编码
- **禁止使用纯黑纯白**：背景色 #090a0f～#12131a，文字 #e2e8f0（珍珠白）/ #94a3b8（板岩灰）
- **像素级毛玻璃规范**：所有卡片/浮动按钮统一 backdrop-blur:12px，边框 1px border-white/5
- **Hover 微光扩散**：蓝紫(#4f46e5)或橙(#ff6b35)微光，扩散半径 ≤20px

### 13.2 动效的性能防线
- **Scanner 列表禁止 GSAP**：必须用 Vue `<TransitionGroup>` 轻量淡入
- 复杂 GSAP 滚动动画只能用在 Dashboard 首页和 Methodology 背书页

### 13.3 极简代码架构
- 每个 Vue 组件自包含样式，禁止套娃超过 3 层
- 禁止出现 "// 后端对接留空" 或 "/* 样式自行补充 */"

## 十四、自动化文档动态同步铁律（2026-07-05 追加）

### 14.1 改完即登（实时日志同步）
每个 Task 验证通过 + git commit 后，**最后一个回复必须附带 PROJECT_STATUS.md 更新**，包括：
- 已完成 Phase 进度条
- 最新改动文件 + 代码行数变化
- 最新 Git Commit Hash

### 14.2 规则迭代（随时追补纪律）
开发过程中讨论并增加/修改的任何规则，**必须立刻写入 WORKFLOW_RULES.md**，不得只留在聊天记录里。

### 14.3 无缝衔接（新对话唤醒）
新对话时第一条消息必须是"读取 PROJECT_STATUS.md 和 WORKFLOW_RULES.md"，通过这两个文件在 1 秒内恢复全部记忆。

## 十六、Phase D 前端开发纪律沉淀（2026-07-05 锁定）

### 16.1 前置反问触发条件
每次开始一个新的前端组件开发前，必须先执行 9.1 反问机制（确认涉及章节、配色、动画参数），禁止直接输出代码。

### 16.2 前端合规检查清单
- [ ] 背景色使用 `#090a0f`～`#12131a`，禁止 `bg-black`/`#000000`
- [ ] 文字使用 `#e2e8f0`（主）/ `#94a3b8`（次），禁止 `#ffffff`
- [ ] 毛玻璃卡片：`backdrop-blur:12px` + `border-white/5`
- [ ] Hover 微光：蓝紫 `#4f46e5` 或橙 `#ff6b35`，spread ≤20px
- [ ] CTA 按钮：用 `<CtaButton>` 组件，传 `color` prop 而非硬编码
- [ ] 路由守卫：已锁死 `beforeEach` 逻辑，禁止绕过 Landing
- [ ] Three.js / ECharts：禁止全局 import，只在对应组件内按需引入
- [ ] 每个 Task 完成后同步更新 PROJECT_STATUS.md + WORKFLOW_RULES.md

### 16.3 改完即登
每个 Task 完成后必须同时提交：
1. PROJECT_STATUS.md（更新进度 + Commit Hash）
2. WORKFLOW_RULES.md（规则迭代沉淀）


## 十七、Bug 修复铁律 (2026-07-06 追加)

1. 一次只改一个 Bug，禁止顺手修
2. 先定位根因（终端/浏览器/log 证据），禁止猜测式修复
3. 一次只改一个文件、一个函数、一行原因
4. 改完立刻验证：PASS → commit，FAIL → 回退
5. 禁止在失败代码基础上继续修——必须回退重来
6. 每次修复前创建 Checkpoint
